# Events Parquet Converter

`events_parquet_to_csv.py` converts parquet files from `data_raw/events/` to the raw CSV format expected by the existing TGN preprocessing flow, analogous to `reddit.csv` and `wikipedia.csv`.

It also uses optional preprocessing artifacts from `data_raw/preprocess/` when they are present.

## Input

By default the converter reads:

```bash
data_raw/events/events_v1.parquet
```

The detected schema is:

```text
src, dst, timestamp, event_type, dt_bucket, has_media, url_count_bucket,
char_count_bucket, url_hash, domain
```

The converter maps:

```text
src -> user
dst -> item
timestamp -> timestamp
label -> state_label, default 0.0
```

The names `user` and `item` are inherited from the original JODIE/TGN CSV format. They do not
necessarily mean a human user and a product/item in this project. For Telegram data:

```text
user -> source endpoint, usually source channel
item -> destination endpoint, usually destination channel
```

The training code expects these column names/positions, so the converter keeps them as a compatibility
format even when the real-world entities are Telegram channels.

Numeric columns not used as endpoints/timestamp are written as edge features. `event_type` is one-hot encoded by default.

## Feature Encoding

The converter produces edge-level features because this implementation of TGN consumes features per
interaction from `data/ml_<name>.npy`.

Event columns are handled as follows:

```text
dt_bucket -> numeric edge feature
has_media -> numeric edge feature
url_count_bucket -> numeric edge feature
char_count_bucket -> numeric edge feature
event_type -> one-hot categorical edge feature
url_hash -> ignored by default
domain -> ignored by default
```

For `event_type`, one-hot columns are generated by pandas. With the current data this gives:

```text
event_type_FORWARD
event_type_SAME_URL
```

Example:

```text
FORWARD  -> event_type_FORWARD=1.0, event_type_SAME_URL=0.0
SAME_URL -> event_type_FORWARD=0.0, event_type_SAME_URL=1.0
```

The high-cardinality textual columns `url_hash` and `domain` are intentionally skipped. They should
not be directly one-hot encoded at this scale. If they become important, use a controlled strategy
such as top-K domains, hashing trick features, or a separate embedding pipeline.

## Preprocess Artifacts

When `data_raw/preprocess/` exists, the converter uses:

```text
data_raw/preprocess/channel_filter_v1.csv
data_raw/preprocess/channel_stats.parquet
data_raw/preprocess/style_features.parquet
```

`channel_filter_v1.csv` is applied by default: only events where both `src` and `dst` have
`kept=True` are exported. Disable this with:

```bash
python3 utils/events_parquet_to_csv.py --no-channel-filter
```

`channel_stats.parquet` and `style_features.parquet` are appended by default as edge features for
both endpoints. Feature names are prefixed internally as `src_*` and `dst_*` before they are written
to numeric feature columns in the raw CSV. Disable this with:

```bash
python3 utils/events_parquet_to_csv.py --no-channel-features
```

Use another preprocessing directory with:

```bash
python3 utils/events_parquet_to_csv.py --preprocess-dir path/to/preprocess
```

The channel-level preprocessing columns are expanded onto every event. For each interaction, the
raw event receives:

```text
event features
event_type one-hot features
src_* channel statistics and style features
dst_* channel statistics and style features
```

For example, `style_features.parquet` columns such as `hour_entropy`,
`night_ratio_utc_00_05`, `delta_mean_s`, `delta_std_s`, `delta_cv`, `has_media_ratio`,
`url_msg_ratio`, `url_per_msg_mean`, `text_len_mean`, and `text_len_std` are included twice:
once as `src_<column>` and once as `dst_<column>`.

`channel_stats.parquet` columns such as `total_messages`, `forward_messages`, `forward_ratio`,
`url_messages`, `url_ratio`, `span_days`, `avg_msgs_per_day`, `first_ts`, and `last_ts` are handled
the same way.

This is a pragmatic adapter for the current TGN codebase. It turns static channel descriptors into
edge features because the existing training code does not load meaningful per-node features from the
raw CSV. The generated `data/ml_<name>_node.npy` remains a zero matrix unless the preprocessing code
is extended further.

## Output Format

The output CSV has the same raw layout as the public JODIE datasets:

```text
user,item,timestamp,state_label,feature_0,feature_1,...
```

By default the raw converted CSV is written under `data_raw/`. This keeps source and intermediate
conversion output separate from the `data/` directory consumed by training.

This file can then be passed to `utils/preprocess_data.py`, which creates:

```text
data/ml_<name>.csv
data/ml_<name>.npy
data/ml_<name>_node.npy
```

## Basic Usage

Convert the full `events_v1.parquet` file:

```bash
python3 utils/events_parquet_to_csv.py --sort
python3 utils/preprocess_data.py --data events --input-dir data_raw --output-dir data --bipartite
```

## Converter Flags

`--shared-node-space` changes how endpoint ids are encoded before the raw CSV is written.

Without this flag, `src` and `dst` are encoded independently:

```text
src channel A -> user id 10
dst channel A -> item id 25
```

With `--shared-node-space`, both columns share one id namespace:

```text
src channel A -> user id 10
dst channel A -> item id 10
```

Use `--shared-node-space` when `src` and `dst` are the same kind of entity, for example Telegram
channels. When this flag is used, run `utils/preprocess_data.py` without `--bipartite`.

`--sort` sorts interactions by timestamp before writing the raw CSV. This is important for temporal
models: the train/validation/test split in `utils/data_processing.py` is based on timestamp
quantiles, and TGN memory updates are easiest to reason about when events are stored in chronological
order. The converter uses a stable sort, so events with the same timestamp keep their original order.

## Using Split Windows

The `splits_*.json` files contain timestamp bounds. Use `--split-file` to export only a selected window:

```bash
python3 utils/events_parquet_to_csv.py \
  --split-file data_raw/events/splits_5M_v1.json \
  --output data_raw/events_5M.csv \
  --sort

python3 utils/preprocess_data.py \
  --data events_5M \
  --input-dir data_raw \
  --output-dir data \
  --bipartite
```

Available split files include:

```text
data_raw/events/splits_v1.json
data_raw/events/splits_10M_v1.json
data_raw/events/splits_5M_v1.json
data_raw/events/splits_2_2M_v1.json
```

To export the full dataset, omit `--split-file`:

```bash
python3 utils/events_parquet_to_csv.py \
  --output data_raw/events_full.csv \
  --sort

python3 utils/preprocess_data.py \
  --data events_full \
  --input-dir data_raw \
  --output-dir data \
  --bipartite
```

## Node Encoding Modes

Default mode encodes `src` and `dst` separately, which is compatible with `--bipartite` preprocessing:

```bash
python3 utils/events_parquet_to_csv.py --output data_raw/events.csv --sort
python3 utils/preprocess_data.py --data events --input-dir data_raw --output-dir data --bipartite
```

`--bipartite` is an option of `utils/preprocess_data.py`, not of the parquet converter. It tells the
original TGN preprocessing code to treat the first CSV column (`user`) and the second CSV column
(`item`) as two disjoint node sets. Internally, it shifts all destination ids by `max(user) + 1` and
then shifts all node ids by one because index `0` is reserved as padding/empty id in the TGN data
arrays.

This is correct for the original JODIE-style datasets:

```text
wikipedia: user -> page
reddit: user -> subreddit/post/thread-like item
```

In such data, a source id `42` and destination id `42` do not mean the same real object. They belong
to different namespaces, so separating them is necessary.

If `src` and `dst` should represent one shared node space, use:

```bash
python3 utils/events_parquet_to_csv.py \
  --shared-node-space \
  --output data_raw/events.csv \
  --sort

python3 utils/preprocess_data.py --data events --input-dir data_raw --output-dir data
```

For Telegram data, if both `src` and `dst` are channel ids, `--shared-node-space` plus preprocessing
without `--bipartite` is usually the cleaner graph semantics. In that case, channel `1001` is the
same node whether it appears as a source or a destination. Using `--bipartite` would duplicate that
channel into two roles: source-channel `1001` and destination-channel `1001`.

Recommended Telegram baseline with the 5M split:

```bash
python3 utils/events_parquet_to_csv.py \
  --shared-node-space \
  --split-file data_raw/events/splits_5M_v1.json \
  --output data_raw/events_5M.csv \
  --sort

python3 utils/preprocess_data.py --data events_5M --input-dir data_raw --output-dir data
```

Recommended Telegram baseline with the full dataset:

```bash
python3 utils/events_parquet_to_csv.py \
  --shared-node-space \
  --output data_raw/events_full.csv \
  --sort

python3 utils/preprocess_data.py --data events_full --input-dir data_raw --output-dir data
```

## Smoke Test

For a quick schema check and small sample:

```bash
python3 utils/events_parquet_to_csv.py \
  --max-rows 100 \
  --print-schema \
  --output data_raw/events_sample.csv \
  --sort

python3 utils/preprocess_data.py \
  --data events_sample \
  --input-dir data_raw \
  --output-dir data \
  --bipartite
```

If channel filtering is enabled, a small sample can contain fewer than `--max-rows` rows because the
limit is applied while reading the parquet file, then rows outside `channel_filter_v1.csv` are removed.

## Custom Columns

If a parquet file uses different column names, override them explicitly:

```bash
python3 utils/events_parquet_to_csv.py \
  --input custom.parquet \
  --source-col source_id \
  --destination-col target_id \
  --timestamp-col event_time \
  --label-col label \
  --output data_raw/custom_events.csv
```

Feature columns can also be selected manually:

```bash
python3 utils/events_parquet_to_csv.py \
  --feature-cols dt_bucket has_media \
  --categorical-feature-cols event_type \
  --output data_raw/events.csv
```

Pass `--categorical-feature-cols` with no values to disable categorical features.

## Critical Notes for Telegram Data and TGN

This conversion is useful, but it is not a perfect semantic match for the original datasets used by
Rossi et al.'s TGN implementation.

The original `wikipedia` and `reddit` datasets are naturally bipartite user-item interaction graphs.
Telegram forwarding and same-url events are closer to channel-channel graphs. If `src` and `dst`
refer to the same kind of entity, `--shared-node-space` is conceptually cleaner than the default
`--bipartite` path. The default path is mainly kept for compatibility with the original preprocessing
assertions and examples.

Static channel style features can leak future information if they were computed over the full time
range and then attached to train events. For strict temporal evaluation, recompute channel statistics
using only information available before each split boundary, or at least before the training cutoff.

The current label is `0.0` when no label column is provided. That is fine for self-supervised link
prediction because labels are not the supervision target there, but it is not meaningful for the
supervised node-classification script. A supervised Telegram task needs an explicit target definition.

`FORWARD` and `SAME_URL` are not equivalent relations. Encoding them as one-hot edge features lets
the model see the relation type, but the graph is still trained as one temporal interaction stream.
For richer relation modeling, a relational temporal model or separate relation-specific sampling may
be better.

High-cardinality text-derived fields such as `domain` and `url_hash` are currently ignored. This is
safer than exploding the feature space, but it also drops potentially important signal. Add them only
through bounded encodings.

Finally, feature scale matters. The converter writes raw numeric values. Counts, timestamps, and
time deltas can have very different magnitudes from binary indicators. For stable training, consider
normalizing continuous channel features before conversion or adding normalization inside the
converter.

## Requirements

The converter requires:

```text
pandas
numpy
pyarrow or another pandas-compatible parquet engine
```

The project README already requires `pandas` and `numpy`; install `pyarrow` if parquet support is missing.
